## 1. 概述

### 1.1 文档目的

本文档是 OmniKey Vault 的**安全实现圣经**,服务于:

- **密码学实现者**:遵循算法参数、密钥生命周期、不变量。
- **代码审查者**:逐条核对威胁防御是否在代码中落地。
- **外部审计人员**:验证 [项目手册.md §7 威胁模型摘要](./项目手册.md#72-威胁模型摘要) 与本文档每一条防御是否生效。
- **漏洞响应者**:判定漏洞等级与缓解优先级。

### 1.3 核心安全主张

> **就算同步过程或云端备份文件整体泄露,只要主密码不丢,攻击者拿到密文也无法在合理时间内还原明文。**
>
> 而要做到这一点,不仅要密码学强度足够,还需要**自定义的、不被通用工具(KeePass / hashcat / John)识别的信封格式**。

### 1.4 安全设计原则

1. **零知识**:主密码、MK、KEK、DEK 永不出设备;服务端(或云盘)只见密文。
2. **抗误用**:所有密码学通过 `ICryptoProvider` 高层 API,禁止业务层直接调用底层库。
3. **最小暴露**:敏感数据在内存中窗口期最小化,用完立即清零。
4. **深度防御**:即使某一层失效(如 DEK 泄露),其他层(如 Profile 隔离)仍能限制损害。
5. **可审计**:所有密码学原语公开、参数明示、代码开源,接受外部审查。
6. **失败安全**:任何异常路径都不应留下明文残留;崩溃时不生成含密钥的转储。

## 2. 威胁模型

### 2.1 我们防御的威胁

> 用户面向摘要见 [项目手册.md §7.2](./项目手册.md#72-威胁模型摘要)。本节为技术详细版本,服务于密码学实现者与外部审计人员。

#### T1:同步云盘被入侵,攻击者拿到 `.okv` 文件

- **攻击者能力**:获取 `vault.okv` 完整密文;可任意次重放 / 修改 / 删除。
- **防御手段**:
  - 客户端零知识加密:主密码永不出设备,KDF 在本地执行。
  - Argon2id 强 KDF(t=3, m=256MiB, p=4):离线爆破成本极高。
  - AEAD 认证加密(XChaCha20-Poly1305):篡改即检测。
  - 容器 Ed25519 签名:外部伪造文件即验签失败。
- **残留风险**:攻击者持有密文可无限期尝试爆破主密码;缓解靠 KDF 强度。
- **验证**:测试用例 `SEC-T1-01` — 模拟云盘文件落地,验证客户端拒绝未签名文件。

#### T2:攻击者用 hashcat / John / KeeCracker 等通用工具离线爆破

- **攻击者能力**:拥有 hashcat 等 GPU 爆破工具与常见密码库。
- **防御手段**:
  - **自定义信封格式**:私有 magic `OKV1`、非默认 Argon2id 参数、私有字段布局。
  - hashcat 等工具的 KDBX4 模式(13400)、Bitwarden 模式均不识别 `.okv`。
  - 攻击者需先逆向我们的代码才能拼出正确的 KDF 调用,显著提高攻击成本。
- **残留风险**:开源后攻击者可获得代码;缓解靠 v1.6 Double Argon2id 密钥拉伸（每次猜测需 320 MiB 内存，成本翻倍）+ 每年小幅迭代字段布局 + KDF 参数动态调整。
- **验证**:测试用例 `SEC-T2-01` — 用 hashcat 常见模式尝试 `.okv`,验证全部失败。

#### T3:设备被植入恶意软件,运行时内存被 dump

- **攻击者能力**:拥有 SYSTEM / 管理员权限,可读取进程内存。
- **防御手段**:
  - 进程内最小暴露:MK / KEK / DEK 仅在解锁窗口期驻内存。
  - 自动锁定:空闲 15 分钟 / 系统锁屏立即锁定,主动清零。
  - 安全擦除:`CryptographicOperations.ZeroMemory` 清零密钥缓冲。
  - 不参与崩溃转储:`SetUnhandledExceptionFilter` 注册清零并吞掉转储的处理器。
- **不防御**:拥有 SYSTEM 权限的实时恶意软件(键盘记录器、内存取证)。这是合理边界,见 §2.2。
- **验证**:测试用例 `SEC-T3-01` — 锁定后扫描进程内存,验证无 MK 残留。

#### T4:同事 / 家人短暂接触设备

- **攻击者能力**:物理接触已解锁设备,短时间内操作。
- **防御手段**:
  - 自动锁屏:系统锁屏事件(`SessionSwitch`)触发立即锁定。
  - 空闲锁定:15 分钟无操作锁定。
  - 可选生物识别二次确认:Windows Hello 仅作为便利解锁,**不**作为唯一因子。
- **验证**:测试用例 `SEC-T4-01` — 模拟系统锁屏,验证 Vault 立即锁定。

#### T5:备份介质(U 盘 / 网盘)丢失

- **攻击者能力**:获取 `vault.okv` 或 `seed.okv.dev` 副本。
- **防御手段**:
  - 生产文件 `vault.okv`:主密码保护,丢失不等于泄露。
  - 种子文件 `seed.okv.dev`:明确警告"等同于明文分发",`--strip-secrets` 模式可剥离 sensitive 字段。
  - 主密码永不写入设备:无 `settings.json` 中的密码字段,无"记住密码"选项。
- **残留风险**:`seed.okv.dev` 丢失即泄露 dev 数据;缓解靠强警告 + 默认 strip-secrets。
- **验证**:测试用例 `SEC-T5-01` — 验证 `seed.okv.dev` 导出时 sensitive 字段已被剥离。

#### T6:社交工程(诱导主密码)

- **攻击者能力**:通过钓鱼、冒充客服等手段诱导用户输入主密码。
- **防御手段**:
  - 不显示明文密码输入框的"上次输入" / 自动填充。
  - 无云端找回:忘记主密码只能用本地 Recovery Key,无"发送重置邮件"。
  - UI 永不请求主密码除非用户主动操作(解锁 / 修改密码)。
- **不防御**:用户主动告知他人主密码。
- **验证**:测试用例 `SEC-T6-01` — 验证无任何"找回密码"入口。

#### T7:重放攻击:旧版本 `.okv` 被回放

- **攻击者能力**:通过同步盘版本回滚,用旧 `vault.okv` 覆盖新版本。
- **防御手段**:
  - 向量时钟单调递增:启动时检测本地 `vector_clock < 远端`,提示用户。
  - 远端向量落后本地:直接拒绝覆盖。
  - 客户端签名:回放的旧文件签名有效但版本落后,合并逻辑处理。
- **残留风险**:用户被诱导接受回滚;缓解靠 UI 明确警告。
- **验证**:测试用例 `SEC-T7-01` — 模拟向量回滚,验证客户端拒绝。

#### T8:供应链攻击:某次同步引入篡改过的文件

- **攻击者能力**:在同步路径上注入篡改后的 `vault.okv`。
- **防御手段**:
  - AEAD 认证加密:XChaCha20-Poly1305 MAC 验证密文完整性。
  - 客户端 Ed25519 签名:验证文件来自可信设备密钥。
  - 构建哈希记录:文件头含 App Build Hash,反供应链溯源。
- **验证**:测试用例 `SEC-T8-01` — 篡改 1 字节,验证客户端拒绝。

### 2.2 我们不防御的威胁(明确告知用户)

| 威胁 | 理由 |
|---|---|
| 设备上已运行的 SYSTEM / 管理员级恶意软件 | OS 边界之外,任何本地优先工具都无法防御 |
| 用户主密码被窃(钓鱼、肩窥、密码本被拍照) | 密码学无法防御人为失误 |
| 物理接触设备 + 已知主密码 | 这是合理的合法访问 |
| 未来量子计算对 XChaCha20-Poly1305 的影响 | 预留后量子升级路径,见 §12 |

### 2.3 威胁优先级矩阵

| ID | 可能性 | 影响 | 防御优先级 |
|---|---|---|---|
| T1 | 高(云盘泄露常见) | 严重 | P0 |
| T2 | 中(攻击者需技能) | 严重 | P0 |
| T3 | 低(需已植入恶意软件) | 严重 | P1 |
| T4 | 中(短暂接触常见) | 中 | P1 |
| T5 | 中(介质丢失常见) | 严重 | P0 |
| T6 | 中(社工普遍) | 严重 | P1 |
| T7 | 低(需云盘版本回滚) | 高 | P2 |
| T8 | 低(需 MITM 同步) | 严重 | P1 |

## 3. 密码学套件

### 3.1 算法选型

| 用途 | 算法 | 参数 | 理由 |
|---|---|---|---|
| 密钥派生 | **Argon2id** | t=3, m=256 MiB, p=4(默认);首次解锁后测量耗时,自动调整到 ≥500ms | RFC 9106,抗 GPU / ASIC,内存硬化 |
| 对称加密 | **XChaCha20-Poly1305** | 每条记录独立 24 字节随机 nonce | RFC 8439 + draft-irtf-cfrg-xchacha;nonce 长度 24B 降低碰撞风险 |
| 认证 / 完整性 | Poly1305 MAC(AEAD 内置) | 128 bit | AEAD 自带,无需独立 MAC |
| 随机数 | OS CSPRNG(`BCryptGenRandom` / `getrandom`) | — | OS 级熵源,抗预测 |
| (可选)第二因子 | WebAuthn / FIDO2 私钥派生恢复码 | 仅用于 Recovery Key 解封 | 抗钓鱼,硬件绑定 |
| 密钥包装(KWrap) | XChaCha20-Poly1305 | 包装 KEK → DEK | 复用 AEAD,简化审计 |
| 容器签名 | Ed25519(本地设备密钥) | 32B 私钥 / 32B 公钥 / 64B 签名 | 防止外部文件伪造;快、小、安全 |

### 3.2 参数选择 Rationale

#### 3.2.1 Argon2id 参数

| 参数 | 值 | RFC 9106 推荐 | 我们选择理由 |
|---|---|---|---|
| `t`(迭代次数) | 3 | 1-3 | 3 次迭代增加 GPU 并行成本 |
| `m`(内存) | 256 MiB | 64 MiB+ | 远超 libsodium 默认 2^19=0.5 MiB;让 hashcat 默认模式失败 |
| `p`(并行) | 4 | 1-4 | 利用多核,同时增加并行爆破难度 |
| `salt` | 32B(随机) | ≥16B | 防彩虹表 |

**动态调整**:首次解锁后测量耗时,若 <500ms,自动将 `m` 翻倍直到 ≥500ms(上限 1 GiB,避免 OOM)。调整后参数写入 `.okv` 头,下次解锁按新参数。

**为什么不直接用 libsodium 默认参数**:
- libsodium `crypto_pwhash` 默认 `m=64MiB, t=3`,内存成本偏低。
- 我们的 256 MiB 让消费级 GPU(8-16GB VRAM)难以并行多实例爆破。
- 同时,非默认参数意味着 hashcat 等工具的预设模式无法直接套用。

#### 3.2.2 Double Argon2id 密钥拉伸（v1.6+ Header v2）

> **v1.6 新增**:为支持公开源码仓库（GitHub public repo），引入双层 KDF 链，将离线爆破成本翻倍。

**威胁背景**:当源码公开后，攻击者可获取完整的 KDF 算法和参数。虽然 Kerckhoffs 原则保证安全性不依赖算法保密，但增加每轮猜测的计算成本可以进一步抬高爆破门槛。

**v1 KDF 链**（HeaderVersion=1，v1.0-v1.5 创建的金库）:
```
MK  = Argon2id(password, salt1[0:16], {t=3, m=256MiB, p=4})
KEK = HKDF-SHA256(MK, "okv-kek-v1", salt1[0:16])
```

**v2 KDF 链**（HeaderVersion=2，v1.6+ 创建的金库）:
```
MK1 = Argon2id(password, salt1[0:16], {t=3, m=256MiB, p=4})   // Round 1: 全量内存成本
MK2 = Argon2id(MK1,     salt2[16:32], {t=3, m=64MiB,  p=1})   // Round 2: 使用 MK1 作为输入的二次拉伸
KEK = HKDF-SHA256(MK2, "okv-kek-v2", salt1[0:16])              // 域分隔符升级为 "okv-kek-v2"
```

| 维度 | v1（单轮） | v2（双轮） |
|---|---|---|
| Argon2id 轮次 | 1 | 2 |
| 每轮猜测总内存 | 256 MiB | 256 + 64 = 320 MiB |
| 每轮猜测相对成本 | 1× | ~2× |
| 合法用户解锁耗时 | ~8-12 秒 | ~10-14 秒 |
| 域分隔符 | `okv-kek-v1` | `okv-kek-v2` |
| Salt 槽利用 | 前 16B（KDF salt），后 16B（保留） | 前 16B（Round 1 salt），后 16B（Round 2 salt） |

**设计决策**:
1. **第二轮使用 MK1 作为 Argon2id 输入**: MK1 是 32 字节的高熵密钥材料，使用它作为 Argon2id 的 "password" 输入是合法的（Argon2id 接受任意长度字节序列）。这确保攻击者必须先完成 Round 1 才能尝试 Round 2，无法跳过。
2. **第二轮参数降低至 64 MiB**: 避免合法用户解锁耗时翻倍过多。64 MiB 仍然远超 libsodium 默认值，且攻击者需对每个猜测都付出此成本。
3. **域分隔符升级**: 从 `"okv-kek-v1"` 变为 `"okv-kek-v2"` 确保 v1 和 v2 的 KEK 不可互换，即使密码相同也无法用 v1 KEK 解密 v2 金库。
4. **二进制布局不变**: Salt 槽仍为 32 字节，后 16 字节从"保留"升级为 Round 2 salt。v1 金库的二进制格式完全兼容，无需迁移。
5. **向后兼容**: `UnlockAsync` 通过 `HeaderVersion` 字段自动选择正确的 KDF 路径。v1 金库继续使用单轮路径，v2 金库使用双轮路径。

**安全保证**: 即使攻击者拥有完整的源代码和 .okv 文件，不知道主密码就无法解密。攻击者必须对每个密码猜测运行两轮 Argon2id（共 320 MiB 内存），这使得离线爆破成本极高。

#### 3.2.2 XChaCha20-Poly1305 而非 AES-GCM

| 维度 | XChaCha20-Poly1305 | AES-GCM |
|---|---|---|
| 硬件加速 | 软件实现快且恒定 | 需 AES-NI,无硬件时慢且时序泄露 |
| Nonce 长度 | 24 字节(碰撞风险极低) | 12 字节(大量加密需维护计数器) |
| 抗误用 | Nonce 随机生成即可 | Nonce 重用即灾难性 |
| 平台一致 | 全平台一致性能 | 无 AES-NI 的 ARM / 旧设备慢 |

**决策**:XChaCha20-Poly1305 的 24 字节 nonce 允许我们随机生成 nonce 而无需维护计数器,降低误用风险。

#### 3.2.3 Ed25519 设备签名

- **用途**:每个设备生成独立 Ed25519 密钥对,私钥用于签名 `.okv` 文件,公钥存于文件头。
- **目的**:防止外部攻击者注入伪造文件(即使他们知道我们的加密格式,也无设备私钥)。
- **生命周期**:设备密钥对绑定于设备,不同步;新设备首次同步时,其公钥被其他设备接受(需用户在 UI 确认)。

### 3.3 密码学套件不变量

1. **不接受字符串**:`ICryptoProvider` 所有方法接受 `ReadOnlySpan<byte>`,调用方负责 UTF-8 编码。
2. **nonce 唯一性**:每次加密使用 OS CSPRNG 生成 24 字节 nonce;不允许复用。
3. **AAD 绑定**:加密时 AAD 绑定文件头元数据(version、vault_uuid、entry_id),防跨上下文重放。
4. **零知识**:任何网络 / 同步路径上不传输 MK / KEK / DEK / 主密码。
5. **签名覆盖**:Ed25519 签名覆盖文件头除签名字段外的所有字节。

## 4. 密钥层级

### 4.1 密钥派生流程

```
主密码 (Master Password, 用户记忆, 不落盘)
    │
    ▼ Argon2id (per-vault salt, t=3 m=256MiB p=4)
MK — 主密钥 (Master Key, 32 字节, 内存中)
    │
    ▼ HKDF-SHA256(info="okv-kek-v1", salt=vault_uuid)
KEK — 密钥加密密钥 (Key Encryption Key, 32 字节, 内存中)
    │
    ├────► 包装每个 Profile 的 DEK (Data Encryption Key)
    │       (Profile Key Blob 存于 .okv 头)
    │
    ├────► 包装每个设备的 DeviceKey (用于签名)
    │
    └────► 派生对称认证密钥 (用于 .okv 文件级认证)
                │
                ▼ XChaCha20-Poly1305 + 设备 Ed25519 签名
            加密的条目集合 (Entries)
```

### 4.2 各密钥的属性

| 密钥 | 长度 | 来源 | 存储位置 | 生命周期 | 销毁方式 |
|---|---|---|---|---|---|
| 主密码 | 变长 | 用户输入 | 仅内存(用户记忆) | 解锁窗口期 | 用户关闭应用 |
| MK | 32B | Argon2id(主密码, salt) | 内存 | 解锁窗口期 | `ZeroMemory` + 锁定 |
| KEK | 32B | HKDF-SHA256(MK, info) | 内存 | 解锁窗口期 | `ZeroMemory` + 锁定 |
| DEK | 32B(每 Profile) | OS CSPRNG 生成 | `.okv` 头(KEK 包装) | Vault 生命周期 | 删除 Profile 时清零 |
| DeviceKey | 32B(Ed25519 私钥) | OS CSPRNG 生成 | 设备本地 keystore(KEK 包装) | 设备生命周期 | 删除设备时清零 |
| Recovery Key | 32B | OS CSPRNG 生成 | 用户离线保存(打印 / 文件) | Vault 生命周期 | 用户销毁 |

### 4.3 关键性质

- **MK 永不落盘**:只在用户已解锁的状态下驻内存。
- **KEK 由 MK 派生**:可被销毁 / 重建;换主密码只需重加密 KEK,无需重加密条目。
- **DEK 由 KEK 包装**:`.okv` 头部存 `WrappedDek = XChaCha20-Poly1305(KEK, DEK, nonce)`。
- **修改主密码 = 重新包装 DEK**:O(1) 操作,不需要重新加密所有条目。
- **Profile 隔离**:不同 Profile 的 DEK 完全独立,一个泄露不影响其他。

### 4.4 密钥生命周期状态机

```
            create
              │
              ▼
       ┌──────────────┐
       │  InMemory    │ ◄──── unlock(主密码)
       │  (MK/KEK/DEK)│
       └──────────────┘
              │
              ├── idle 15min ──► ┌──────────────┐
              │                  │   Locked     │
              ├── SessionSwitch ──► (MK/KEK/DEK  │
              │                  │  zeroed)     │
              ├── user lock ──────► └──────────────┘
              │
              └── app exit ──► process teardown
```

### 4.5 Recovery Key 机制

- **生成**:创建 Vault 时,OS CSPRNG 生成 32 字节随机数,展示为 32 组 4 字符 base32 分组(便于抄写)。
- **存储**:用 KEK 包装后存于 `.okv` 头的 `RecoveryBlob` 字段;明文由用户离线保存(打印 / 加密 U 盘)。
- **使用**:用户输入 Recovery Key → 解包 `RecoveryBlob` 得到 KEK → 重建 MK 上下文(仅用于重置主密码,不能作为常规解锁因子)。
- **强制**:v1 强制要求首次创建 Vault 时导出 / 打印 Recovery Key(见 [项目手册.md §19 待定问题 2](./项目手册.md#19-待定问题))。
- **轮换**:用户可在设置中轮换 Recovery Key(生成新 key + 重新包装)。

## 5. 自定义信封格式设计哲学

### 5.1 设计目标

> 让通用工具(hashcat / John / KeeCracker)"看不懂" `.okv` 文件,提高攻击成本。

### 5.2 关键设计选择

| 维度 | 通用格式做法 | 我们的做法 | 效果 |
|---|---|---|---|
| Magic bytes | `0x9AA2D903` (KDBX) 等公开值 | `OKV1` (0x4F 4B 56 01) 私有 | 通用工具不识别 |
| KDF 参数 | libsodium 默认 64 MiB | 256 MiB + 自定义版本号字段 | hashcat 预设模式失败 |
| 字段布局 | 公开规范 | 私有布局 + 版本号字段 | 需逆向代码才能拼装 KDF 调用 |
| 签名 | 无或可选 | 强制 Ed25519 设备签名 | 外部伪造即拒收 |

### 5.3 为什么不直接用 KDBX4 / Bitwarden 格式

- **KDBX4**:公开标准,hashcat mode 13400 专门支持,泄漏后自动识别并爆破。
- **Bitwarden**:vault JSON 结构公开(通过逆向),GPG / 对称加密封装参数固定。
- **我们的 `.okv`**:即使攻击者知道我们用了 Argon2id,也需要先逆向代码才能拼出正确的 KDF 调用 → 显著提高攻击成本。

### 5.4 开源与攻击成本的权衡

| 因素 | 影响 |
|---|---|
| 开源代码 | 降低攻击成本(攻击者可读代码) |
| 公开规范文档 | 降低攻击成本 |
| 非默认 KDF 参数 | 提高攻击成本(通用工具失败) |
| 私有 magic + 字段布局 | 提高攻击成本(需逆向) |
| 每年迭代字段布局 | 提高攻击成本(逆向需持续投入) |
| 强 KDF(256 MiB) | 提高攻击成本(单次爆破成本高) |

**结论**:开源带来的"审计可信"收益 > 攻击成本降低的损失。我们靠强 KDF + 持续迭代维持攻击成本,而非靠闭源。

详见 [文件格式规范.md](./文件格式规范.md) 的二进制布局规范。

## 6. 内存安全

### 6.1 敏感数据类型

| 类型 | 来源 | 内存形态 |
|---|---|---|
| 主密码 | 用户输入 | `byte[]`(UTF-8),用完立即 `ZeroMemory` |
| MK / KEK / DEK | 派生 / 解包 | `Memory<byte>` 封装于 `SecureKey : IDisposable` |
| 明文 Entry | 解密 | `byte[]`(JSON UTF-8),反序列化后立即清零 |
| 剪贴板内容 | 复制 | `byte[]`,8 秒后清零 |

### 6.2 安全内存管理

- **`SecureKey` 类型**:封装 `Memory<byte>`,`Dispose` 调用 `CryptographicOperations.ZeroMemory`。
- **`using` 模式**:所有密钥对象在 `using` 块中使用,确保离开作用域即清零。
- **PIN**:关键场景(主密码处理)使用 `fixed` / `GCHandle.Alloc(.., Pinned)` 防止 GC 移动后留下残留副本。
- **禁止字符串**:敏感数据不使用 `string`(不可变 + 实习池 + 无法清零);全部 `byte[]` / `Span<byte>`。

### 6.3 锁定行为

- **自动锁定**:空闲 15 分钟(可配置 5-60 分钟)。
- **系统锁屏**:订阅 `Microsoft.Win32.SystemEvents.SessionSwitch`(Windows),`SessionSwitchReason.SessionLock` 触发立即锁定。
- **锁定动作**:
  1. `CryptographicOperations.ZeroMemory(MK / KEK / DEK)`
  2. 清空 Entry 缓存
  3. 清空搜索索引缓存
  4. UI 切换到解锁界面
  5. 释放文件锁

### 6.4 不参与崩溃转储

```csharp
// Windows 实现
SetUnhandledExceptionFilter(exception => {
    ZeroAllKeys();
    FlushLog();
    return EXCEPTION_EXECUTE_HANDLER; // 吞掉转储
});
```

- **目的**:防止崩溃转储文件包含 MK 等密钥。
- **代价**:调试困难,但可通过 `--enable-crashdump` 开发模式覆盖(仅 DEBUG 构建)。

### 6.5 不写临时文件

- 导出:使用内存流直接写入用户指定位置,无中间临时文件。
- 解密:解密结果在内存流中,不落地磁盘。
- 例外:原子写入用 `.okv.tmp`,但该文件是密文,且写入完成立即 rename 删除。

### 6.6 内存安全不变量

1. 所有 `SecureKey` 实例必须 `using` 或显式 `Dispose`。
2. `string` 不得承载主密码 / MK / KEK / DEK / 明文 Entry。
3. 锁定时所有 `SecureKey` 必须 `ZeroMemory`。
4. 崩溃处理器必须先清零再退出。

## 7. 侧信道缓解

### 7.1 时序侧信道

| 操作 | 缓解 |
|---|---|
| 主密码验证 | 恒定时间比较(`CryptographicOperations.FixedTimeEquals`) |
| 签名验证 | Ed25519 验证内部恒定时间 |
| MAC 验证 | Poly1305 验证恒定时间 |

### 7.2 内存侧信道

- **缓存时序**:XChaCha20 软件实现恒定时间(无查表);AES 在无 AES-NI 时有缓存时序,故选 XChaCha20。
- **分支泄露**:密码学核心路径无数据依赖分支。
- **GPU 侧信道**:不适用(无 GPU 计算)。

### 7.3 电源 / 电磁侧信道

- **不防御**:面向消费级笔记本 / 工作站,不防御实验室级侧信道(成本不匹配)。

### 7.4 软件侧信道(Spectre / Meltdown)

- **不防御**:依赖 OS 级缓解(KPTI / Retpoline),应用层无法独立防御。

## 8. 密钥轮换与主密码修改

### 8.1 主密码修改流程

```
1. 用户输入旧主密码 → 验证(verifyTag)
2. 用户输入新主密码(两次) → 强度检查
3. 派生新 MK' = Argon2id(新主密码, 同一 salt)  // 保持 salt 不变
   (可选:生成新 salt,但需重新包装所有 DEK)
4. 派生新 KEK' = HKDF(MK', "okv-kek-v1")
5. 对每个 Profile:重新包装 DEK
   wrapped_dek' = XChaCha20-Poly1305(KEK', DEK, new_nonce)
6. 原子写入 .okv(含新 verifyTag、新 wrapped_dek')
7. 清零旧 MK / KEK
```

- **复杂度**:O(Profiles),与条目数无关。
- **条目不重新加密**:DEK 不变,仅重新包装。
- **崩溃恢复**:若步骤 6 崩溃,`.okv.tmp` 残留,下次启动检测并清理,旧 `.okv` 仍可用旧主密码。

### 8.2 DEK 轮换

- **触发**:用户主动("重置 Profile 密钥")或安全事件响应。
- **流程**:
  1. 生成新 DEK'
  2. 重新加密该 Profile 所有条目(O(Entries))
  3. 包装 DEK' 存入 `.okv`
  4. 清零旧 DEK
- **性能**:1 万条目约 5-10 秒。

### 8.3 设备密钥轮换

- **触发**:设备丢失 / 怀疑泄露,用户在另一设备吊销该设备公钥。
- **流程**:
  1. 从 `manifest.json` 的 `devices` 列表移除该设备
  2. 重签名 `.okv`(用当前设备私钥)
  3. 其他设备同步后拒绝接受已吊销设备的签名

### 8.4 Recovery Key 轮换

- **流程**:
  1. 生成新 Recovery Key(32B CSPRNG)
  2. 用当前 KEK 包装存入 `RecoveryBlob`
  3. 用户展示新 Key,提示销毁旧 Key
- **注意**:旧 Recovery Key 在用户销毁前仍有效(无法强制远程销毁)。

## 9. 恢复密钥 (Recovery Key)

### 9.1 设计目标

- **唯一离线恢复手段**:主密码丢失时,用 Recovery Key 重建访问。
- **不作为常规解锁因子**:避免成为"第二主密码"被频繁使用。
- **抗暴力破解**:32B 随机数,无法爆破。

### 9.2 格式

```
Recovery Key = 32 字节随机数
展示形式:32 组 4 字符 base32,分隔符 "-"
例:ABCD-EFGH-IJKL-MNOP-QRST-UVWX-YZ23-4567-...
```

- **校验位**:末尾附 2 字符 CRC-8 校验,防抄写错误。
- **展示场景**:首次创建 Vault 时强制展示,用户必须打印或保存到加密介质。

### 9.3 使用流程

```
1. 用户在解锁页选择"使用 Recovery Key"
2. 输入 32 组字符(带校验)
3. 客户端用 Recovery Key 解包 RecoveryBlob 得到 KEK
4. 用 KEK 重建 MK 上下文(仅用于重置主密码)
5. 强制用户立即设置新主密码
6. 用新主密码派生新 MK / KEK,重新包装 DEK
7. 生成新 Recovery Key(旧 Key 失效)
```

### 9.4 安全声明

- **Recovery Key 等同于主密码**:拥有 Recovery Key 即拥有完全访问权限。
- **用户责任**:安全保存 Recovery Key,丢失即无法恢复。
- **无云端找回**:Recovery Key 永不传输到任何服务器。

## 12. 已知限制与未来改进

### 12.1 后量子(PQ)升级路径

- **现状**:`OKV1` 格式非 PQ 安全。
- **风险**:量子计算对 XChaCha20-Poly1305 的威胁(10+ 年远期)。
- **路径**:v2 引入混合 PQ 加密(XChaCha20-Poly1305 + ML-KEM 包装),`OKV2` magic。
- **迁移**:v2 客户端可读 `OKV1`(自动升级),v1 客户端遇到 `OKV2` 提示升级。

### 12.2 生物识别的边界

- **Windows Hello**:仅作为便利解锁(替代输入主密码),**不**作为唯一因子。
- **原因**:生物特征不可轮换,泄露即永久;且 OS 级生物识别的安全性由 OS 保证,我们无法审计。

### 12.3 多用户共享

- **不支持**:零知识模型的硬约束(无解密就无共享)。
- **替代**:`seed.okv.dev --strip-secrets` 用于团队 dev 数据流转。

### 12.4 团队 / 企业功能

- **不支持**:RBAC、SSO、审计日志转发。
- **建议**:企业需求接 Vaultwarden / Bitwarden Enterprise。

## 13. 漏洞响应流程

### 13.1 报告渠道

- **Security Bug Bounty**:security@omnikeyvault.example(待定)
- **PGP 公钥**:公布于 `/security/.well-known/pgp.txt`
- **响应 SLA**:确认收到 ≤48 小时;初步评估 ≤7 天;修复发布 ≤90 天(按严重度)。

### 13.2 严重度分级

| 等级 | 定义 | 示例 | 响应 |
|---|---|---|---|
| Critical | 可远程无需交互获取明文 | AEAD 误用导致密文可还原 | 7 天内修复 + 紧急发布 |
| High | 可在特定场景获取明文 / 密钥 | 内存未清零导致 dump 泄露 MK | 30 天内修复 |
| Medium | 削弱安全不变量但不直接泄露 | KDF 参数被错误调低 | 90 天内修复 |
| Low | 理论威胁 / 加固建议 | UI 信息泄露 | 下个版本修复 |

### 13.3 披露策略

- **协调披露**:报告者 + 维护者协商发布时间。
- **默认 90 天**:报告后 90 天公开,除非修复需要更久且报告者同意延期。
