## [2.3.2] - 2026-07-22 — 修复更新检查误报"已是最新版本"

### 🐛 修复

#### 检查更新时网络错误/速率限制被误报为"已是最新版本"
- **根因**：`UpdateService.CheckForUpdateAsync()` 的 `catch` 块静默吞掉了所有异常（网络超时、GitHub API 速率限制 403、DNS 解析失败等），统一返回 `null`。调用方将 `null` 解读为"无更新"，导致用户看到"✓ 已是最新版本"而实际上检查根本没成功
- **修复**：引入 `UpdateCheckResult` 结构化返回类型，区分三种状态：
  - `NoUpdate` — 当前版本已是最新
  - `UpdateAvailable` — 发现新版本
  - `CheckFailed` — 检查失败（附错误信息）
- **修复**：手动检查更新时，若检查失败则显示"✕ 检查失败: {具体原因}"（如速率限制、网络错误等），不再误报"已是最新版本"
- **修复**：启动时自动检查仍然静默失败（不打扰用户），但用户可从设置页面手动检查获取错误详情
- **修复**：分类捕获异常（`TaskCanceledException` → 超时、`HttpRequestException` → 网络错误、其他 → 通用错误），提供有意义的错误信息

### ✅ 测试
- 全部 791/791 测试通过（781 原有 + 10 新增 v2.3.2 测试），无回归
- 新增 `V23UpdateCheckTests`（10 项测试）：覆盖 `UpdateCheckResult` 三种状态的标志位、版本比较、安装包识别

## [2.3.1] - 2026-07-22 — 修复浏览器扩展目录未包含在安装包中

### 🐛 修复

#### 安装后找不到 browser-extension 目录
- **根因**：`browser-extension/` 目录存在于项目根目录，但 csproj 未配置将其复制到发布输出目录，导致 `dotnet publish` 和 Inno Setup 安装包均不包含该目录。用户按照设置界面说明尝试加载扩展时，找不到 `browser-extension/` 目录
- **修复**：在 `OmniKeyVault.Cli.csproj` 中添加 `browser-extension/` 目录的 `CopyToOutputDirectory` 规则，确保所有扩展文件（manifest.json、popup.html、popup.js、README.md）自动包含在发布输出中
- **修复**：更新设置界面的安装说明，明确指出 `browser-extension/` 文件夹位于程序安装目录下（默认 `C:\Program Files\OmniKeyVault\browser-extension`），便携版位于解压目录下
- **修复**：更新使用手册第 18.3 节，补充安装版和便携版的具体路径

## [2.2.2] - 2026-07-22 — 自包含发布，消除 .NET 运行时依赖

### 🐛 修复

#### 安装 .NET 后仍提示需要安装/更新
- **根因**：应用使用框架依赖模式（`--self-contained false`）发布，`runtimeconfig.json` 只引用了 `Microsoft.NETCore.App`，但 Avalonia UI 需要的 `Microsoft.WindowsDesktop.App` 未被包含。用户即使安装了 .NET SDK 或 .NET Runtime，如果安装的不是 **Desktop Runtime** 版本，应用就无法启动并提示需要安装/更新 .NET
- **修复**：改为**自包含发布模式**（`--self-contained true`），将完整的 .NET 运行时（`coreclr.dll`）和 Windows Desktop 程序集（`WindowsBase.dll` 等）直接打包进应用目录。用户安装后无需单独安装任何 .NET 运行时，彻底消除此类问题
- **额外保障**：在 `csproj` 中添加 `<RollForward>LatestMinor</RollForward>`，确保应用可以运行在任何 8.x 补丁版本上

### ⚠️ 注意
- 安装包体积从 ~10 MB 增加到 ~45 MB（自包含模式包含完整 .NET 运行时），这是确保零运行时依赖的必要代价
- 便携版体积从 ~11 MB 增加到 ~80 MB

### ✅ 测试
- 全部 730/730 测试通过，无回归

## [2.2.1] - 2026-07-21 — 单例模式运行 + 托盘图标修复

### ✨ 新增

#### 单例模式运行
- **单实例锁定**：应用现在以单例模式运行，同一时间只允许一个进程实例。如果用户尝试启动第二个实例，已有实例的主窗口会自动弹出到前台，第二个实例立即退出
- **进程间通信**：使用命名 Mutex 检测 + 命名管道 IPC 实现，第二个实例通过管道向第一个实例发送"SHOW"命令，第一个实例收到后将窗口激活到前台
- **应用退出清理**：应用退出时自动释放 Mutex 和管道服务器资源

### 🐛 修复

#### 托盘图标显示空白
- **根因**：`EnsureTrayIcon` 中的 `AssetLoader.Open` 在某些发布构建中静默失败（异常被空 catch 吞掉），导致 `WindowIcon` 为 null，托盘图标显示为空白
- **修复**：重写图标加载逻辑 `LoadTrayIcon`，采用多重策略：
  1. **策略 1（优先）**：从磁盘上的 `.ico` 文件加载（`Assets/okv-icon.ico`）
  2. **策略 2（备选）**：从 AvaloniaResource 嵌入资源加载
  3. 每个策略都带有详细的诊断日志
- **额外保障**：`.ico` 文件现在同时作为 `AvaloniaResource`（嵌入程序集）和 `None/CopyToOutputDirectory`（拷贝到输出目录）双重打包，确保策略 1 在发布版本中始终可用
- **流生命周期**：将原始流复制到 `MemoryStream`，避免 `using` 提前释放导致图标数据丢失

- 全部 730/730 测试通过，无回归

## [1.6.0] - 2026-07-12 — Double Argon2id 密钥拉伸（公开源码安全加固）

### 🔒 安全升级
- **Double Argon2id 密钥拉伸（Header v2）**: 新增双层 KDF 链，为公开源码场景提供更强防护。
  - **Round 1**: `MK1 = Argon2id(password, salt1, {256 MiB, t=3, p=4})` — 与 v1 相同的全量内存成本
  - **Round 2**: `MK2 = Argon2id(MK1, salt2, {64 MiB, t=3, p=1})` — 使用 MK1 作为输入的二次拉伸
  - **KEK**: `HKDF-SHA256(MK2, "okv-kek-v2", salt1)` — 域分隔符从 `okv-kek-v1` 升级为 `okv-kek-v2`
  - 攻击者每次密码猜测需运行两轮 Argon2id（共 320 MiB 内存），爆破成本翻倍
  - 合法用户解锁仅增加 ~1-2 秒（64 MiB 第二轮开销）
- **Salt 槽复用**: 32 字节 salt 槽的前 16 字节为 Round 1 KDF salt，后 16 字节从"保留"升级为 Round 2 KDF salt（二进制布局不变）
- **向后兼容**: v1 金库（HeaderVersion=1）继续使用单轮 Argon2id 路径，v2 金库（HeaderVersion=2）使用双轮路径

### 🐛 修复
- **ChangePasswordAsync 数据丢失 bug**: `ChangePasswordAsync` 错误地从 `_vault.Profiles`（创建/解锁时的快照）读取 Profile 数据，而非 `_profiles`（工作副本），导致在 `PutEntry` 后直接调用 `ChangePasswordAsync` 会丢失新添加的条目。已修正为使用 `_profiles`。

### ✅ 测试
- 新增 17 项 Double Argon2id KDF 专项测试（`DoubleArgon2KdfTests`），覆盖：
  - KDF 基本属性（确定性、不同密码/salt 产生不同密钥）
  - v1/v2 KEK 不可互换性
  - 端到端金库生命周期（创建/解锁/数据完整性）
  - 向后兼容性（v1 金库可解锁、错误密码拒绝）
  - 修改密码往返（v2 金库）
  - 安全不变量（源码泄露 + .okv 文件 ≠ 解密；v1 KDF 无法解锁 v2 金库）
  - Save/Load 保留 Header 版本
- 全部 578/578 测试通过（561 已有 + 17 新增）。
